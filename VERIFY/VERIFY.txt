ORL-Chat Verification
Deterministic Bounded Conversation-State Resolution

Governing relation:

same deduplicated supported message fragments + same resolver rules
-> same bounded conversation-state result


1. Verification Scope

The repository provides three separate verification activities:

A. Browser visual review
   - visually presents the committed scenario
   - shows different arrival presentations and a scripted resolution sequence
   - does not independently execute the full Python resolver
   - does not independently enumerate all 120 permutations

B. Python reference-scenario checking
   - executes the bounded resolver
   - compares the two node states before and after sharing
   - checks all 5! = 120 permutations of the committed five-message scenario
   - computes deterministic result hashes

C. Frozen artifact-identity checking
   - compares committed file bytes against SHA-256 values
   - proves file identity only
   - does not by itself prove behavioral correctness

A successful frozen-hash comparison establishes artifact identity only.

It does not establish that the implementation behaves correctly.

Regenerate frozen hashes only after intentional file changes and after completing the behavioral checks.


13. Pass Criteria

The committed reference scenario passes when all of the following are observed:

- the Python script loads the committed input
- no unexpected exception occurs on the committed input
- Node-A is INCOMPLETE before sharing
- Node-B is INCOMPLETE before sharing
- the nodes differ before sharing
- both nodes hold equal deduplicated supported evidence after sharing
- both nodes produce equal post-sharing resolver states
- the final state is RESOLVED
- the final message identifier is M4
- the final declared value is Meeting at 5 PM
- the final text corresponds to the committed M4 fragment
- checked_permutations = 120
- permutation_independence = True
- the state-signature hash is produced
- the replay certificate is produced
- repeated unchanged runs reproduce the same bounded result
- frozen artifact hashes match when the compared files are intentionally unchanged


14. Fail Criteria

The committed reference scenario fails review if any of the following occurs:

- the committed input cannot be loaded
- an unexpected exception occurs on the committed input
- either node is not INCOMPLETE before sharing
- the nodes incorrectly match before sharing
- the post-sharing node states differ
- the final state is not RESOLVED
- the final message identifier is not M4
- the final declared value is not Meeting at 5 PM
- the final text does not correspond to the committed M4 fragment
- fewer or more than 120 unique-fragment permutations are checked
- any tested permutation produces a different bounded state signature
- repeated unchanged runs produce materially different bounded results
- a frozen artifact hash fails without an intentional file change


15. What a Passing Result Establishes

A passing result establishes that the current Python implementation, for the committed supported scenario:

- loads the documented fragments
- produces the documented pre-merge and post-merge states
- produces same-evidence node equality after scripted sharing
- selects the declared value Meeting at 5 PM
- reproduces the same bounded state across all 120 committed-fragment permutations
- produces deterministic hashes for the current serialized result object

These are bounded implementation and scenario results.

ORL-Chat is verified at the current bounded reference-scenario level when:

- two initially incomplete nodes receive the same deduplicated supported five-message fragment set
- both produce the same RESOLVED conversation state
- the final message identifier is M4
